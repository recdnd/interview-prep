本頁は **題庫本体ファイルに含めない** 運用メモ。`[xx]` 本文は統合後 **`packs/smart-full-lines.js`** の題名接頭 **`〔hard-work〕`** セクション（旧 `hard-work` id 1〜6 相当）。  
Cursor や人間が `[xx]` を差し替えるときの**記録規範**と、貼り付け用の**短い指示文**を置く。

## Cursor にそのまま渡す指示（短）

```text
この pack 内容は **`smart-full-lines`**（`〔hard-work〕` 題）内の `window.PACK` として扱ってください。
`[xx]` は差し替え可能なプレースホルダです。初期値は必ず `（例: ...）` 付きで残してください。
差し替え時は、プレースホルダの役割に合う語だけを入れ、文の論理骨格は崩さないでください。
各 variant は単独で読める完全な文にしてください。What / Why / Example を別 variant に分割しないでください。
`role` は補助的な内部用です。主説明は title と本文（text）で行ってください。
```

## `[xx]` プレースホルダ規約（記録用）

- 文中の `[xx]` は、**差し替え可能なプレースホルダ**である。
- 各 `[項目名]` には、初期状態で **1 つデフォルト値**を入れる。形式は **`[項目名]（例: デフォルト値）`** で統一する。
- Cursor で文面を更新するときは、`[項目名]` 自体は残してもよいし、運用上不要なら**デフォルト値だけを残す形**に置換しても構わない。
- ただし、意味の骨格までは変えず、**「何を直そうとして、途中で何に気づき、なぜそちらを優先したか」**という流れは維持する。
- `[xx]` を差し替えるときは、**抽象度を揃える**。例：`[機能名]` には機能名、`[根本原因]` には原因、`[将来の悪影響]` には将来コストを入れる。
- 1 つの question の `variants` は、**同じ結論・同じ効き方の平行表現**として保つ。What / Why / Example を別 variant に分割しない。
- 短句の寄せ集めにせず、**各 variant は単独で読んでも意味が通る完全な文**にする。
- `role` は内部管理用。読み手への主説明は **`title` と `text`** で行う。
- 新規 pack や registry 変更時は **`npm run check-packs`** を忘れない。

## `PACK_FORMAT.md` との関係

- 題庫の一般規則（variant の独立性、`role` の位置づけなど）は **[PACK_FORMAT.md](./PACK_FORMAT.md)** に従う。
- **`[xx]（例: …）`** という書き方は、上記規則を満たしたうえで、**差し替え可能欄**を明示するためのこのプロジェクト内慣習として本メモで定義する。
