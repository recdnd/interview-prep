# david-pich-routing

給 **Cursor／自己** 用的第一版「整場控制器」：對齊倉庫裡現有 pack，不另造題號幻覺。

---

## 目標

- 讓對方覺得是**同一種 problem framing** 的人，但不滑進抽象哲學閒聊。
- 最後落到**可以一起做產品**（能接話、能問回去、能收束）。

---

## 全局 flow（主路徑）

與現有題組對齊後的建議主線：

`START`  
→ **id 5** 自己紹介（位置定義）  
→ **id 6** 強み（對齊對方語境）  
→ **id 7** Ruvia（入口）  
→ **id 8** 相性（橋接對方身份／文脈）  
→ **id 10**（＋必要時 **id 9**）應用・具体・AI バランス  
→（有興趣）**id 11** demo  
→（後半）**`ml-research-pack`**：問對方、把球丟回去  
→ **id 13** 降密度（聽＋短接球）  
→ **id 14** closing  

開場若要暖場，可在 **id 5** 前插入 **id 1**／**id 2**／**id 12** 擇一～二，避免全上拖長。

## Phase 分解

### Phase 1：同類感

- 使用：**id 1**、**id 2** 或 **id 12**（擇需）、**id 5**
- 目標：不像「只會考試的學生」，而是**同一種整理曖昧的方式**的人。

### Phase 2：能力

- 使用：**id 6**、**id 7**、**id 8**
- 目標：讓人覺得你能跑「曖昧 → 切る → 試す」。
- 必須：**每一段都掛一個小例子**（id **10** 的語感或 demo 題材預告），避免只有抽象詞。

### Phase 3：可合作

- 使用：**id 11**（選擇性）、**`ml-research-pack`**、**id 13**
- 目標：從「有趣的人」→「可以一起丟進 team 的人」。
- **後半場關鍵**：少切 pack、多用 **CBL／CRE relay** 短句，讓對方講。

## 逐題 routing（`david-pich.js`）

### id 5 — 自己紹介（約 30 秒）

- **作用**：定位自己 = problem framing／structure 側。
- **何時**：開場主線第一步。
- **追問後**：
  - 「最近在做什麼」→ **id 7**
  - 「為什麼適合我們」→ **id 4** 或 **id 8**
- **demo**：無  
- **helper**：CCE（要長一點結構時）  
- **avoid**：一開始 Spiral 深講、長過頭。

### id 6 — 強み（最重要）

- **作用**：把能力翻成對方聽得懂的語言。
- **何時**：被問強み／強みっぽい話題。
- **追問後**：
  - 「具體怎麼做」→ **id 7**
  - 「有沒有例子」→ **id 8** 或 **id 10**（＋必要時 **id 11**）
- **demo**：可選  
- **helper**：CBL  
- **if-asked 方向**：「具体的には？」「どうやって？」  
- **avoid**：重複空喊「構造」「曖昧」而無例子。

### id 7 — Ruvia（入口）

- **作用**：thinking を外に出す interface として自分を置く。
- **何時**：「最近在做什么」系問題。
- **追問後**：
  - 和一般筆記差別 → **`ruvia-development-explain` id 4**
  - 怎麼用／工程感 → **id 10** 或 **`ruvia-development-explain` id 5～8** 擇近
  - 為何對方（DS／研究側）會有感 → **id 8**
- **demo**：有 Ruvia 就接 **id 11** 的說法  
- **helper**：CBL  
- **avoid**：一上來 deep spec-first、太哲學。

### id 8 — 相性（橋接）

- **作用**：對齊對方身份與敘事（記事・分析・意思決定の質）。
- **何時**：對方開始有興趣、或要你談「為何我們」。
- **追問後**：
  - 應用／產品 → **id 10**
  - 想看 → **id 11**
- **demo**：可搭配 Ruvia／題材一句  
- **helper**：CBL  
- **avoid**：把對方窄化成「只會 ML 的工程師」、完全不碰業務文脈。

### id 10 — 具体・「抽象だけじゃない」

- **作用**：把你拉到「做完什麼」的產品／検証側。
- **何時**：「那用在哪」「實際怎麼落地」。
- **追問後**：
  - demo → **id 11**
  - 業務／組織 → **`ml-research-pack`**（問對方）
- **helper**：CBL  
- **avoid**：只抽象、完全不提 SaaS／現場。

### id 11 — デモ

- **作用**：從「講」到「一起看」。
- **何時**：對方明顯有興趣、時間允許。
- **操作腳本（約 3 分・講＋做）**：**[`ruvia-demo-routing.md`](./ruvia-demo-routing.md)**（trigger、分鐘切分、結束後接話、對齊 `ruvia-development-explain` id 6／10）。
- **next**：demo 後 → **`ml-research-pack`** 或 **id 13** → **id 14**。
- **avoid**：demo 過長、過細到像在炫技。

### id 13 — 後半「降密度」

- **作用**：對方開始長講時，**讓對方舒服**（接球、不追加自我證明段）。
- **何時**：對方輸出變多、或進閒聊／雙向段。
- **next**：少跳 pack；**CBL、CRE relay**。
- **avoid**：再講一大段自己、再堆成就。

### id 14 — closing

- **作用**：收束、禮貌離場。
- **helper**：CCE（若要收束成一段完整締め）。

## 全局注意

### 不要

- 一直 abstract  
- 一直只講 Spiral  
- 一直自我證明  

### 要做

- 每段盡量掛**一個例子**  
- 讓對方接話、讓對方講  
- **後半**：少切 pack → **helper で短文**

不是來「證明最強」，而是來讓對方覺得：**這人可以丟進 team 一起做**。
